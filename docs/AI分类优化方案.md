# AI 智能分类优化方案

> 版本：v1.0 | 日期：2026-05-25

## 一、问题现状

### 1.1 批次间上下文碎片化

当前 AI 分析采用分批处理（默认每批 50 个书签），但每批次**独立分析**，互不感知。

**后果**：
- 批次 1 创建了"AI工具"，批次 2 不知道，又创建了"人工智能/工具"
- 后处理的合并只能处理名称**高度相似**的情况，语义相关但名称不同的分类无法合并
- 用户已有"人工智能/工具"分类时，AI 可能另起炉灶，产生重复分类

### 1.2 父子分类语义丢失

当前 prompt 只传递扁平的路径列表（如 `"人工智能/工具"`），没有体现层级关系。

**后果**：
- AI 不理解"人工智能/工具"是"AI相关的工具"，可能把通用工具放进去
- 分类的语义上下文被截断，父级对子级的约束力丢失

### 1.3 用户无调整能力

当前确认对话框是**纯只读 + 一键全量应用**：
- 没有 checkbox 取消某个分类
- 没有搜索/筛选功能
- 没有重命名能力
- 书签数量多时，信息堆叠混乱

---

## 二、优化目标

| 目标 | 说明 | 优先级 |
|------|------|--------|
| 减少分类碎片化 | AI 能看到前序批次的结果，避免重复创建 | P0 |
| 用户分类优先 | 冲突时优先使用用户已有分类，AI 创建的为辅 | P0 |
| 父子语义约束 | 子分类继承父级的语义上下文 | P1 |
| 冲突检测与提示 | AI 发现语义重叠时主动提示用户决策 | P1 |
| 可交互确认对话框 | 用户能逐条/逐分类勾选、搜索、重命名 | P1 |

---

## 三、方案设计

### 3.1 批次间上下文传递

**改动文件**：`src/api/openai.js`、`src/background/background.js`

#### 设计思路

1. **累积上下文**：每批完成后，将当前批次确定的分类结果（名称 + 书签数 + 来源）传给下一批
2. **分组标注**：标注每个分类的来源（"用户已有" vs "前序批次新建"），让 AI 理解上下文
3. **合并指引**：明确告诉 AI，前序批次新建的分类如果与用户已有分类语义重叠，应合并到用户已有分类

#### 改后的 prompt 结构

```
## 用户现有分类（共 15 个，最高优先级）
- 人工智能/工具
- 人工智能/教程
- 技术/前端
- 技术/后端
...

## 前序批次已确定的分类（供参考）
以下分类是前序批次中 AI 创建的，仅供参考。
如果它们与用户现有分类语义重叠，应优先使用用户现有分类。
- "AI工具" (12个书签) → 建议合并到 "人工智能/工具"
- "AI教程" (5个书签) → 建议合并到 "人工智能/教程"

## 输入数据
（本批次书签列表）
```

#### 实现方式

- `handleAIAnalyze()` 中维护 `accumulatedCategories` 数组
- 每批完成后，将该批确定的分类追加到 `accumulatedCategories`
- 下一批调用 `buildAnalysisPrompt()` 时传入 `accumulatedCategories`

### 3.2 父子分类语义约束

**改动文件**：`src/api/openai.js`、`src/api/prompts/classification-system.prompt.js`

#### 设计思路

将扁平路径列表转换为**层级树结构**输出，体现父子关系：

```
- 人工智能
  - 人工智能/工具（AI工具、平台、应用）
  - 人工智能/教程（AI学习资料、课程）
- 技术
  - 技术/前端（Web前端技术、框架）
  - 技术/后端（服务端、数据库）
```

#### 实现方式

修改 `buildCategoryTree()` 函数，输出层级缩进格式。同时在 prompt 中增加规则：

> 当书签被分配到子分类时，必须确保其内容与父分类的主题相关。
> 例如："人工智能/工具"下的书签必须与 AI 相关，不能是任意工具。

---

### 3.3 冲突检测与提示

**改动文件**：`src/api/prompts/classification-system.prompt.js`、`src/api/openai.js`

#### 设计思路

让 AI 在输出中增加 `conflicts` 字段，主动报告语义重叠的分类：

```json
{
  "categories": [...],
  "tags": [...],
  "conflicts": [
    {
      "aiCategory": "AI工具",
      "suggestedMerge": "人工智能/工具",
      "reason": "语义重叠，AI工具是用户已有分类'人工智能/工具'的子集",
      "bookmarkIds": ["id1", "id2"]
    }
  ]
}
```

#### 处理流程

1. AI 分析完成后返回 `conflicts`
2. background.js 解析 conflicts，在对话框中展示冲突提示
3. 用户可以选择：合并到推荐分类 / 保留 AI 新建的分类 / 取消该分类

---

### 3.4 可交互确认对话框

**改动文件**：`src/popup/popup.js`、`src/popup/popup.css`

#### 布局设计（左右分栏）

```
┌──────────────────────────────────────────────────────┐
│  AI 整理建议                                   [×]   │
├───────────────┬──────────────────────────────────────┤
│ 分类列表       │  选中分类的书签详情                    │
│               │                                       │
│ ☑ 人工智能/工具 (12)  🔍 [搜索书签...]               │
│ ☑ 人工智能/教程 (5)                                   │
│ ☑ 技术/前端 (18)      ☑ 书签1 - url1                 │
│ ⚠ AI工具 → 人工智能/工具  ☑ 书签2 - url2             │
│ ☑ 技术/后端 (8)        ☐ 书签3 - url3                │
│ ☑ 学习/语言 (3)                                       │
│               │                                       │
│ ──────────── │  标签建议                              │
│ 统计: 15/20   │  [AI] [LLM] [教程] [学习]            │
├───────────────┴──────────────────────────────────────┤
│              [取消]         [应用整理方案]             │
└──────────────────────────────────────────────────────┘
```

#### 交互功能

| 功能 | 操作 | 说明 |
|------|------|------|
| 分类级勾选 | 左侧 checkbox | 取消 = 该分类不应用，书签保留原位 |
| 书签级勾选 | 右侧 checkbox | 单个书签取消 = 该书签不移动 |
| 搜索书签 | 右上搜索框 | 按标题/URL 过滤 |
| 冲突提示 | ⚠ 图标 + 高亮 | AI 分类与用户分类语义重叠时提示 |
| 重命名 | 分类名旁编辑图标 | 可修改分类路径 |
| 折叠/展开 | 点击分类名 | 大量书签时默认折叠 |
| 全选/全不选 | 顶部 checkbox | 批量操作 |
| 统计信息 | 底部状态栏 | 已选 X/Y 个分类，X/Z 个书签 |

#### 数据结构

```javascript
// 对话框内部状态
const dialogState = {
  categories: [
    {
      name: "人工智能/工具",
      isNew: false,
      bookmarkIds: ["id1", "id2"],
      enabled: true,        // 用户是否勾选
      bookmarks: [
        { id: "id1", title: "...", url: "...", enabled: true }
      ]
    }
  ],
  conflicts: [
    {
      aiCategory: "AI工具",
      suggestedMerge: "人工智能/工具",
      bookmarkIds: ["id3"],
      resolved: false,       // 用户是否已决策
      action: "merge"        // merge | keep | discard
    }
  ],
  searchQuery: "",
  selectedCatIndex: 0       // 右侧显示哪个分类的书签
};
```

## 六、风险与注意事项

1. **API 调用成本**：批次上下文会增加 prompt 长度，但不会增加调用次数
2. **向后兼容**：`APPLY_CATEGORIES` 消息格式不变，仅增加可选的 `conflicts` 字段
3. **旧版对话框保留**：`ai-analysis.js` 中的旧版对话框暂不改动，避免影响其他功能
4. **分类路径格式**：用户决策后的重命名需要同步更新 IndexedDB 中的分类记录
