# 智能收藏夹管理插件 - 需求设计说明书

## 1. 项目概述

### 1.1 项目名称
Smart Bookmarks - 智能收藏夹管理插件

### 1.2 项目背景
用户在使用浏览器收藏功能时普遍面临以下痛点：
- 历史收藏链接过多、不易归类整理
- 失效链接不易发现和清理
- 缺乏备份机制，换机容易丢失数据

### 1.3 项目目标
开发一款浏览器插件，提供智能分类、失效检测、备份恢复等功能，帮助用户高效管理收藏夹。

### 1.4 目标浏览器
- Chrome (及 Chromium 内核浏览器如 Edge、Brave)
- Firefox (后续支持)

## 2. 功能需求

### 2.1 核心功能模块

#### 2.1.1 数据同步与管理

| 功能 | 描述 |
|------|------|
| 浏览器收藏导入 | 首次使用时，自动导入浏览器现有收藏夹数据 |
| 本地副本维护 | 插件内部维护收藏数据副本，所有操作在本地完成 |
| 确认后同步 | 用户确认操作后，同步更新到浏览器收藏夹 |

#### 2.1.2 AI 智能分类

| 功能 | 描述 |
|------|------|
| 用户自配置 API | 用户自行配置 OpenAI 兼容的 API 地址和模型 |
| 一键分析 | 手动触发 AI 分析所有收藏链接 |
| 现有分类采集 | 优先分析用户已有的文件夹结构，作为分类参考 |
| 智能分类建议 | AI 基于标题、URL、网页描述（如有）提供分类建议 |
| 多标签支持 | 一个链接可归属于多个分类/标签 |
| 汇总确认界面 | 显示分类汇总，用户可查看明细并确认后一键应用 |

#### 2.1.3 失效链接检测

| 功能 | 描述 |
|------|------|
| 一键扫描 | 用户手动触发全量收藏链接检测 |
| HEAD→GET 智能降级 | 优先发 HEAD（4s 超时），遇到 405/5xx/WAF拦截/超时/连接断开 自动降级为 GET 重试 |
| WAF/反爬识别 | 对 401/403/429/520/521/999 等状态码识别为"不确定"(uncertain)，不误判为失效 |
| 失效判断标准 | GET 返回 404/410 等明确错误码视为失效；DNS 解析失败直接判定失效 |
| 三态标记 | `ok`（有效）/ `broken`（失效）/ `uncertain`（WAF拦截，内容可能正常） |
| 待清理文件夹 | 失效链接移动至"待清理"分类，uncertain 链接不移入 |
| 断点续检 | 检测中断后可恢复续检，对话框选择"续检"或"全新检测" |
| 增量持久化 | 每条书签检测完立即写入 DB，中途取消也不丢失已检测结果 |
| 自动恢复 | 重新检测后，原先失效但已恢复的书签自动从"待清理"移回 |
| 一键删除 | 用户确认后可一键删除待清理分类中的链接 |

#### 2.1.4 智能搜索

| 功能 | 描述 |
|------|------|
| 关键词搜索 | 用户输入关键词搜索收藏链接（标题、URL、描述） |
| 高级搜索语法 | `tag:标签名`、`site:域名`、`"精确匹配"`、`排除:关键词` |
| 相关性评分 | 本地搜索显示相关性百分比（0-100%） |
| AI 语义搜索 | 配置 AI 后，支持语义理解搜索，智能排序 |
| 混合搜索 | 本地 + AI 结合，取长补短 |
| 搜索历史 | 自动记录最近 50 条搜索，聚焦搜索框时显示 |
| 搜索缓存 | AI 搜索结果缓存 5 分钟，避免重复请求 |
| 搜索结果高亮 | 高亮显示匹配的关键词和标签 |

#### 2.1.5 备份与恢复

| 功能 | 描述 |
|------|------|
| 本地导出 | 导出为 JSON/HTML 格式文件 |
| 本地导入 | 从备份文件导入数据 |
| 浏览器同步 | 支持浏览器账号自带的收藏同步功能 |

## 3. 技术方案

### 3.1 技术栈

| 技术 | 用途 |
|------|------|
| Manifest V3 | Chrome 扩展标准 |
| Vanilla JavaScript | 核心逻辑（不使用框架以减小体积） |
| IndexedDB | 本地数据存储 |
| Chrome Bookmarks API | 浏览器收藏夹操作 |
| Fetch API | HTTP 请求（AI 调用、链接检测） |

### 3.2 数据存储

| 存储方案 | 用途 |
|----------|------|
| IndexedDB | 收藏数据主存储 |
| chrome.storage.local | 用户配置（API 密钥、偏好设置） |

### 3.3 数据结构设计

#### 收藏项 (BookmarkItem)
```typescript
interface BookmarkItem {
  id: string;                    // 唯一标识
  title: string;                 // 标题
  url: string;                   // 链接
  description?: string;          // 网页描述（content script 采集）
  favicon?: string;              // 网站图标
  tags: string[];                // 标签列表（multiEntry 索引）
  categoryId?: string;           // 所属分类ID
  dateAdded: number;             // 添加时间戳
  dateModified?: number;         // 修改时间戳
  status: 'active' | 'broken';   // 书签状态
  checkStatus?: string;          // 检测状态详情（ok/broken/uncertain/timeout/dns_error/network_error）
  checkError?: string;           // 检测错误描述
  checkStatusCode?: number;      // HTTP 状态码
  lastChecked?: number;          // 上次检测时间戳
  browserId?: string;            // 浏览器收藏ID（用于同步）
}
```

#### 分类 (Category)
```typescript
interface Category {
  id: string;
  name: string;
  parentId?: string;             // 父分类ID（支持层级）
  icon?: string;
}
```

#### 用户配置 (UserConfig)
```typescript
interface UserConfig {
  aiProvider: {
    apiUrl: string;
    apiKey: string;
    model: string;
  };
  preferences: {
    autoSync: boolean;
    theme: 'light' | 'dark' | 'auto';
  };
}
```

### 3.4 网页信息采集方案

采用 **渐进式采集** 策略：

| 场景 | 采集方式 |
|------|----------|
| 新增收藏 | 用户打开页面时，content script 自动采集标题、描述、favicon |
| 历史收藏 | 先用标题+URL 分析，用户可选择"刷新信息"来补充描述 |
| 手动刷新 | 用户右键菜单选择"刷新信息"，打开网页后采集 |

### 4.3 交互流程

#### AI 分析流程
```
用户点击 [一键分析]
    ↓
检查 AI 配置（未配置则提示配置）
    ↓
显示进度弹窗（正在分析 X/Y...）
    ↓
调用 AI API 批量分析
    ↓
显示汇总结果对话框
    ↓
用户点击 [查看明细] 查看具体分类
    ↓
用户点击 [确认并应用]
    ↓
更新本地数据 + 同步到浏览器
```

#### 失效检测流程
```
用户点击 [失效检测]
    ↓
检查是否有中断会话 → 有则询问"续检/全新检测"
    ↓
过滤已检测书签（续检模式下跳过已检测）
    ↓
并发检测链接（并发池 10 个 worker）
    ↓
每个 URL：HEAD(4s) → 失败则降级 GET(6s)
    ↓
每条结果立即写入 DB（增量持久化）
    ↓
    ├── 2xx/3xx → ok（有效）
    ├── 404/410 → broken（失效）→ 移至"待清理"分类
    ├── 521/403 等 WAF 码 → uncertain（不确定）→ 不移入待清理
    ├── DNS 失败 → dns_error（失效）→ 移至"待清理"分类
    └── 超时/网络错误 → timeout/network_error → 移至"待清理"分类
    ↓
原先失效但已恢复的书签自动从"待清理"移回
    ↓
提示用户：X 个失效 + Y 个不确定（WAF 拦截）
```

## 9. 关键技术改进记录

### 9.1 失效链接检测优化（2026-02-28）

**问题**：原始实现使用 HEAD 方法检测，许多正常站点被误判为失效：
- CSDN 等站点部署 WAF，对自动化请求返回 521（embedded_captcha challenge）
- cnblogs 等站点静默丢弃 HEAD 连接，导致超时被判为失效
- 宝塔面板等服务端对 HEAD 实现有缺陷，返回 500

**解决方案**：
1. **HEAD→GET 全面降级**：HEAD 失败（405/5xx/WAF码/超时/网络错误）自动降级为 GET 重试
2. **UNCERTAIN 状态码识别**：401/403/429/520/521/999 标记为 `uncertain`，不视为失效
3. **浏览器特征头**：添加 `Accept`/`Accept-Language` 请求头降低 WAF 拦截率
4. **双超时阈值**：HEAD 4s / GET 6s，HEAD 更短以快速降级
5. **DNS 失败不降级**：域名不可达直接判定失效
